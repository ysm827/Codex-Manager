# apps/src 模块索引

## 目标

本目录是桌面前端主代码区。本文档用于说明各子目录职责，避免后续把页面逻辑、运行时逻辑、设置逻辑继续无序堆回 `main.js`。

## 顶层文件

- `main.js`：前端启动装配入口，负责把各模块组装起来
- `api.js`：前端统一 API / Tauri 调用入口
- `state.js`：共享前端状态容器

## 子目录边界

### `runtime/`

负责：

- 启动初始化编排
- 跨页面刷新流程
- 页面启动后的一次性运行时协同

当前入口：

- `app-bootstrap.js`
- `app-runtime.js`

### `services/`

负责：

- 与后端交互的数据刷新
- service 生命周期管理
- 登录、更新、批量操作等业务流

约束：

- 这里放“做事”的逻辑，不放页面 DOM 渲染细节
- 若是账号、日志、Key 的批量动作，优先继续收敛到 `services/management/`

### `settings/`

负责：

- 设置页状态与事件绑定
- 设置快照、运行配置、环境变量、Web 安全相关逻辑
- 设置项与 service 热同步

当前入口：

- `controller.js`
- `bind-settings-events.js`
- `service-sync.js`

### `views/`

负责：

- 页面渲染
- 视图事件委托
- 表格/卡片/列表的 UI 组织

约束：

- 视图层优先只依赖 render actions，不直接塞大量业务副作用

### `ui/`

负责：

- 通用 UI 控件与反馈组件
- 主题、状态条、按钮 busy、启动遮罩等基础交互能力

### `utils/`

负责：

- 与具体业务弱相关的工具函数
- 格式化、剪贴板、请求工具、代理归一化

### `styles/`

负责：

- 全局样式
- 布局样式
- 组件样式
- 响应式与性能模式样式

## 放置原则

1. 新增页面联动逻辑：优先放 `services/` 或 `runtime/`
2. 新增设置项：优先放 `settings/`
3. 新增渲染与 DOM 结构：优先放 `views/`
4. 新增通用 UI 能力：优先放 `ui/`
5. 新增纯工具函数：优先放 `utils/`

## 当前治理重点

- `main.js` 已开始瘦身，只允许继续保留装配职责
- 设置域已拆成多子模块，后续不要把新逻辑再直接堆回 `settings/controller.js`
- 页面级测试优先放 `apps/tests`，模块级测试优先贴近源码目录
