# 长期维护结构优化建议

## 结论

CodexManager 当前已经具备“可持续开发”的基础形态：前端、桌面端、服务端、发布链路、迁移和测试都已独立成块，不是临时拼装项目。

但如果按长期维护项目的标准看，当前最需要继续治理的不是“功能缺失”，而是以下四类问题：

1. 总控入口文件仍偏大，边界虽然存在，但收口不彻底。
2. 治理文档已经开始形成，但还没有成为稳定的版本化资产。
3. 测试数量不少，但分层命名与回归矩阵还不够一眼可读。
4. 发布、脚本、文档三套体系虽然已经统一方向，但还没有完全形成“新同事一看就明白”的正式工程感。

## 扫描范围与事实基线

### 仓库形态

- 顶层已分出 `apps/`、`crates/`、`scripts/`、`.github/`、`docker/`、`assets/`。
- Rust workspace 只纳入 `crates/core`、`crates/service`、`crates/start`、`crates/web`，桌面端 `apps/src-tauri` 单独维护。
- 已存在基础治理文档：
  - `README.md`
  - `README.en.md`
  - `ARCHITECTURE.md`
  - `CONTRIBUTING.md`
  - `CHANGELOG.md`

### 代码规模感知

- `apps/src/` 下约 `71` 个 JS 文件。
- 前端单测约 `22` 个，UI 结构测试约 `14` 个。
- Rust 文件约 `212` 个，测试相关文件约 `65` 个。
- 说明项目不是“没测试”，问题主要在于治理可读性与结构收敛。

### 明显偏大的入口 / 关键文件

- `apps/src/main.js`
- `apps/src/settings/controller.js`
- `apps/src-tauri/src/lib.rs`
- `crates/service/src/lib.rs`
- `crates/service/src/gateway/observability/http_bridge.rs`
- `crates/service/src/gateway/protocol_adapter/request_mapping.rs`
- `crates/service/src/gateway/protocol_adapter/response_conversion/sse_conversion.rs`
- `.github/workflows/release-all.yml`

## 当前做得对的部分

### 1. 目录主边界已经成立

- 前端 UI、Tauri 壳、服务端、Web 壳、启动器、数据库迁移已分别落在独立目录。
- `ARCHITECTURE.md` 与 `CONTRIBUTING.md` 已经开始承担边界说明，而不是只靠口头传递。

### 2. 后端模块化方向是对的

- `crates/service/src` 已按 `account/`、`apikey/`、`gateway/`、`http/`、`rpc_dispatch/`、`usage/` 拆分。
- `gateway/` 下又分出 `auth/`、`request/`、`routing/`、`upstream/`、`observability/`、`protocol_adapter/`，整体方向正确。

### 3. 测试意识已经具备

- 前端不是只有页面测试，也有结构测试与服务逻辑测试。
- Rust 侧有模块内测试，也有 `crates/service/tests/` 这类更接近集成测试的组织。

### 4. 发布链路已经从“脚本堆叠”走向“单入口”

- 现在已经统一到 `.github/workflows/release-all.yml`。
- `.github/actions/` 也在承接 workflow 复用逻辑，这个方向需要继续坚持。

## 主要问题与优化建议

## P0：先处理治理基础，不然项目很难显得“正式”

### 1. 让 `docs/` 回归版本管理，不要继续默认忽略整个目录

现状：

- 当前 `.gitignore` 仍忽略 `/docs/`。
- 这会导致治理文档虽然能写到本地，但默认不会成为仓库正式资产。

影响：

- 结构治理、排障结论、决策记录无法自然沉淀进仓库主线。
- 容易出现“文档写了，但协作者在仓库里看不到”的问题。

建议：

- 调整为“忽略临时文档、保留正式文档”。
- 例如保留 `docs/plan/`、`docs/decision/`、`docs/release/`，只忽略 `docs/report/*_temp.md` 或临时扫描报告。

### 2. 定义正式的 `docs/` 信息架构

现状：

- 目录规范在协作规则里有说明，但仓库里没有稳定形成常驻文档结构。

建议：

- 固定以下目录含义：
  - `docs/plan/`：实施计划与任务拆解
  - `docs/decision/`：ADR / 决策记录
  - `docs/release/`：发布说明、回滚与验收
  - `docs/report/`：一次性扫描、排障、回归报告
- 在 `docs/README.md` 里说明每类文档是否应提交。

### 3. 把“仓库治理约束”从 README 中进一步抽离成可执行规范

现状：

- `README.md` 已经很长，同时承担介绍、发布、构建、变更入口等多种职责。

建议：

- README 只保留：
  - 项目是什么
  - 快速开始
  - 主要产物
  - 文档入口
- 其余内容分别收敛到：
  - `ARCHITECTURE.md`
  - `CONTRIBUTING.md`
  - `docs/release/*`

## P1：让代码结构更像“正式项目”而不是“能跑的项目”

### 4. 继续拆 `apps/src/main.js` 总控职责

现状：

- `apps/src/main.js` 同时承担启动编排、页面路由、刷新总控、设置桥接、服务生命周期总控。

建议：

- 继续向以下方向拆：
  - `bootstrap/`：启动与初始化
  - `composition/`：模块装配
  - `runtime/`：刷新调度、自动刷新、页面切换协同
- 保持 `main.js` 只做“组装”和“启动”。

### 5. 把设置模块从“控制器 + 绑定”继续收敛成领域对象

现状：

- `apps/src/settings/controller.js` 已经较大，且承担默认值、持久化、格式化、提示、表单同步等多种职责。

建议：

- 继续按设置域拆分：
  - `settings/theme/`
  - `settings/service-bind/`
  - `settings/upstream-proxy/`
  - `settings/background-tasks/`
  - `settings/env-overrides/`
- 每个设置域暴露统一接口：
  - `read`
  - `normalize`
  - `render`
  - `persist`
  - `syncToService`

### 6. 把 `apps/src-tauri/src/lib.rs` 收敛成桌面壳入口，而不是命令全集

现状：

- `apps/src-tauri/src/lib.rs` 虽然已经抽出子模块，但仍保留大量 `#[tauri::command]` 和窗口/托盘生命周期编排。

建议：

- 增加 `commands/` 目录，按领域拆：
  - `commands/account.rs`
  - `commands/service.rs`
  - `commands/settings.rs`
  - `commands/export.rs`
- `lib.rs` 只保留：
  - app builder
  - menu/tray/window wiring
  - command registration

### 7. 让 `crates/service/src/lib.rs` 只做服务启动与公共导出

现状：

- `crates/service/src/lib.rs` 仍承载过多模块声明、常量、运行时配置和设置相关逻辑。

建议：

- 继续分出：
  - `app_runtime/`
  - `settings_keys.rs`
  - `service_bind.rs`
  - `bootstrap.rs`
- `lib.rs` 只保留公共导出、默认常量和最薄的启动入口。

### 8. 给 `gateway/` 再补一层“设计文档级别”的目录说明

现状：

- `crates/service/src/gateway/` 已经是项目最复杂的核心，但只有代码结构，没有目录级别说明。

建议：

- 在 `crates/service/src/gateway/README.md` 或 `docs/decision/` 中补一份：
  - 请求进入顺序
  - route / candidate / upstream / adapter / observability 的关系
  - `/v1/responses` 与 `/v1/chat/completions` 的统一链路
- 这会显著降低后续协议适配维护成本。

### 9. 继续拆超大协议适配文件

重点文件：

- `crates/service/src/gateway/observability/http_bridge.rs`
- `crates/service/src/gateway/protocol_adapter/request_mapping.rs`
- `crates/service/src/gateway/protocol_adapter/response_conversion/sse_conversion.rs`

建议：

- 以“协议 + 流式/非流式 + 错误回退”三个维度拆分。
- 避免单文件既管解析、转换、桥接、统计、错误分类。

## P2：提高测试、脚本、发布体系的可读性

### 10. 统一前端测试目录语义

现状：

- 现在同时存在：
  - `apps/src/**/tests/*.test.js`
  - `apps/tests/*.test.js`

建议：

- 明确两类测试语义：
  - `src/**/tests`：模块级单测
  - `apps/tests`：页面集成/结构回归
- 在 `apps/TESTING.md` 中写清楚执行入口和适用范围。

### 11. 补一份 Rust 侧测试分层说明

现状：

- Rust 侧测试文件不少，但模块内测试、目录级测试、端到端测试的边界没有文档化。

建议：

- 增加 `crates/service/tests/README.md`，说明：
  - 哪些是单元测试
  - 哪些是 HTTP/RPC 集成测试
  - 哪些是协议兼容基线

### 12. 收敛脚本入口命名与平台差异

现状：

- `scripts/` 已经比以前清晰，但 Windows PowerShell、bash、workflow action 之间仍有平行实现。

建议：

- 给 `scripts/README.md` 说明：
  - 本地开发用哪几个
  - 发版前检查用哪几个
  - 哪些脚本只供 workflow 调用
- 降低“看见脚本很多但不知道该用哪个”的问题。

### 13. 给发布链路补“角色边界”

现状：

- `.github/workflows/release-all.yml` 是单入口，但它和 `.github/actions/*`、`scripts/release/*` 的职责边界还不够可读。

建议：

- 在 `docs/release/` 增加发布架构说明：
  - workflow 负责编排
  - composite action 负责公共步骤
  - `scripts/release/*` 负责跨平台细节

## P3：提升正式项目观感

### 14. 增加仓库级 `TESTING.md`

目标：

- 让新协作者快速知道：
  - 改前端跑什么
  - 改 service 跑什么
  - 改协议适配必须跑什么
  - 改 workflow 至少校验什么

### 15. 增加仓库级 `SECURITY.md`

原因：

- 这是对外发布项目的标准治理件之一。
- 至少应该说明：
  - 不接受公开提交敏感 token
  - 如何报告安全问题
  - Web 访问密码、RPC token、账号 token 的基本处理原则

### 16. 增加 issue / PR 模板

原因：

- 现在仓库已经不是纯个人脚本仓库，应该让问题与 PR 输入更规范。

建议：

- `.github/ISSUE_TEMPLATE/bug_report.yml`
- `.github/ISSUE_TEMPLATE/feature_request.yml`
- `.github/pull_request_template.md`

## 推荐治理顺序

### 第一阶段：先修治理基建

1. 恢复 `docs/` 的正式版本管理。
2. 增加 `docs/README.md`、`TESTING.md`、`SECURITY.md`。
3. 收敛 README 的职责边界。

### 第二阶段：拆入口总控文件

1. 拆 `apps/src/main.js`
2. 拆 `apps/src/settings/controller.js`
3. 拆 `apps/src-tauri/src/lib.rs`
4. 拆 `crates/service/src/lib.rs`

### 第三阶段：补核心复杂域文档

1. 补 `gateway/` 链路说明
2. 补测试分层说明
3. 补发布架构说明

## 最终目标

项目不需要为了“像正式项目”而过度企业化。

但至少应该达到下面这个状态：

- 新协作者 10 分钟内能看懂目录边界。
- 新功能落点不会优先堆到总入口。
- 文档能稳定进入仓库主线，而不是停留在本地。
- 发布、测试、设置、协议适配都有固定入口和明确责任人视角。

只要把这几件事做完，CodexManager 的工程观感会明显从“持续迭代的强功能项目”进一步提升到“结构清楚、可长期维护的正式项目”。
