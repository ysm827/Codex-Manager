# Codex 源码对齐差异说明

生成时间：2026-03-16

## 范围

本说明对比以下两类实现：

- 上游 Codex 源码：`D:\MyComputer\own\GPTTeam相关\codex`
- 当前 CodexManager 实现：`D:\MyComputer\own\GPTTeam相关\CodexManager\CodexManager`

本次仅做源码差异说明，不改业务代码。

对比重点是：

- `codex-rs/app-server` 的协议面
- `codex-rs/core` 的传输、鉴权、计划类型、错误模型
- `codex-rs/login` 的登录闭环
- 当前 CodexManager 的 `service/rpc/gateway/auth`

不在本次范围内的内容：

- Codex TUI/CLI 交互细节
- 桌面界面视觉差异
- 已经按产品策略主动偏离 Codex 的行为，不视为“遗漏”，但会单独标注

## 未支持或明显没对齐

### 1. App Server 协议面仍未完整实现

上游 Codex 的主能力面不是一个“管理接口集合”，而是一个完整的双向 JSON-RPC app-server 协议，包含：

- `initialize` + `initialized` 握手
- `thread/start`、`thread/resume`、`thread/fork`
- `turn/start`、`turn/interrupt`
- `thread/compact/start`
- `review/start`
- `thread/realtime/start`
- `item/*`、`turn/*`、`thread/*`、`account/*` 等持续通知
- 审批请求流

当前 CodexManager 的 RPC 面虽然已补了少量 Codex app-server 兼容方法，但主体仍是本地管理服务，主要只有：

- `account/*`
- `apikey/*`
- `app_settings/*`
- `usage/*`
- `service_config/*`
- `gateway/*`
- `requestlog/*`

差异结论：

- 如果目标是“兼容 Codex app-server 协议”，当前实现仍不兼容。
- 当前的 `initialize` / `initialized` 只对齐了最小握手入口，不具备官方“每连接初始化状态机 + 初始化前拒绝其他方法”的完整语义。

依据：

- 上游：[codex-rs/app-server/README.md](../../../../codex/codex-rs/app-server/README.md)
- 当前：[crates/service/src/rpc_dispatch/mod.rs](../../crates/service/src/rpc_dispatch/mod.rs)

## 有意偏离 Codex 的当前实现

以下行为是当前产品为可用性做的主动偏离，不应误判为“没做”：

1. `free / 7天单窗口` 账号统一按设置模型改写请求。
   这不是上游 Codex 的已知行为，而是 CodexManager 为多账号路由场景做的可用性策略。

2. 请求日志记录“首尝试账号”和“尝试链路”。
   这是为了网关排障增强的产品特性，不是 Codex app-server 协议要求。

3. 多账号候选、自动 failover、free/team 混合调度。
   当前产品定位决定了它更偏“代理与账号池管理器”。

## 影响判断

如果目标是“把 CodexManager 做成 Codex app-server 的兼容实现”，优先级最高的缺口是：

1. app-server 双向协议和事件流
2. Responses WebSocket / prewarm / reuse
3. review / realtime / skills / plugins / apps / MCP 深能力面
4. account/rate limits 通知流
5. 登录和错误模型的细节对齐

如果目标只是“让 Codex CLI/请求链路更像 Codex 的上游 HTTP 行为”，优先级最高的是：

1. responses websocket / prewarm / reuse
2. 更细的登录错误解析
3. compact 之外的 memory / summarize 能力
4. 每连接初始化状态机与事件流

## 本次结论

一句话总结：

当前 CodexManager 已经把“HTTP 代理关键 on-wire 行为 + 一批 app-server 基础方法面”补到了可用，但距离 Codex 的 app-server + core client 体系，仍有一整层事件流、WebSocket 传输、线程/回合生命周期和运行时通知协议没有实现。
