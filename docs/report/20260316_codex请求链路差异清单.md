# Codex 请求链路差异清单

生成时间：2026-03-16

## 范围

本清单只覆盖真正影响下面这条链路的部分：

- `Codex / Codex 风格客户端`
- `CodexManager`
- `chatgpt.com/backend-api/codex`

只看这些内容：

- 登录与 token
- `/v1/responses`
- `/v1/responses/compact`
- 请求头、压缩、会话锚点、失败诊断
- `responses websocket / prewarm / reuse` 是否值得进入主线

明确不看：

- `thread/*`
- `turn/*`
- `review/*`
- `skills/*`
- `plugin/*`
- `app/*`
- 本地 app-server / RPC 兼容层

## 当前可保留的已对齐项

### 登录与账号基础

- 浏览器登录主干参数已基本对齐：
  - `/oauth/authorize`
  - PKCE
  - `state`
  - `redirect_uri`
  - `id_token_add_organizations=true`
  - `codex_cli_simplified_flow=true`
  - `allowed_workspace_id`
  - `originator`

- 授权码换 token 的请求体已对齐：
  - `grant_type=authorization_code`
  - `code`
  - `redirect_uri`
  - `client_id`
  - `code_verifier`

- `id_token -> openai-api-key` 的 token exchange 请求体已对齐

- 默认 `issuer / client_id / originator` 基线已对齐官方默认值

### `/v1/responses`

- 普通 `/responses` 仍走 HTTP + SSE 主链路，和当前产品形态一致
- 已透传并维护这些关键会话信号：
  - `prompt_cache_key`
  - `session_id`
  - `x-client-request-id`
  - `x-openai-subagent`
  - `x-codex-turn-state`

- 已支持动态：
  - `Originator`
  - `User-Agent`
  - `x-openai-internal-codex-residency`

- 已支持对流式 `/responses` 做 `zstd` 请求体压缩

- 已有 challenge / 401 / 429 / failover / route quality / 请求日志落盘的主链路

### `/v1/responses/compact`

- 已走真实上游 `/responses/compact`
- 已按 compact 专用 allowlist 和头部语义发请求
- 已按非流式 JSON unary 请求处理

## 当前主要差异

### P0. 登录启动与回调错误建模仍偏弱

当前差异：

- 登录启动是“失败放行”
  - 本地 callback server 没起来时，仍会返回 `auth_url`
  - 用户会看到“能登录”，但回调一定落空

- 回调错误只做了最小处理
  - 对 `error / error_description` 没有完整建模
  - `state mismatch`
  - `workspace restriction`
  - `token_exchange_failed`
  这些和官方相比都还偏弱

- 没有做 workspace 二次校验
  - 目前只是把 `allowed_workspace_id` 放进 authorize URL
  - 没有在 callback 成功后再核验一次绑定的 workspace/account

影响：

- 直接影响登录成功率
- 会造成“看起来成功、实际上失败”或“绑定到错误 workspace”的误判

建议优先级：

- 最高

### P1. refresh / plan type 仍有误判空间

当前差异：

- refresh 是当前自建链路，不是沿官方登录链复用下来的统一实现
- refresh 失败是否把账号标成 `inactive`，仍主要依赖错误字符串匹配
- `planType` 在 refresh 返回路径上可能回退旧 `id_token`，导致套餐识别滞后或掉成 `unknown`

影响：

- 会误摘账号
- 会误识别 free / go / team / pro
- 会间接影响模型选择、候选过滤和路由策略

建议优先级：

- 最高

### P2. `/v1/responses` 主链路还需要继续做 on-wire 复核

当前已做得比较接近，但仍应继续核对这些点：

- 请求体字段白名单与默认值
- stream / non-stream 分支上的 header profile
- cookie、turn state、conversation 相关头在不同链路上的带法
- 失败重试、failover、日志落盘时机

说明：

- 当前不是“完全没对齐”
- 而是还缺一轮严格的 on-wire 收口

影响：

- 直接影响成功率
- 直接影响 challenge 命中概率
- 也会影响日志可解释性

建议优先级：

- 高

### P3. `/v1/responses/compact` 主体应保留，但只补远端压缩真正相关部分

当前差异：

- compact 请求本身已基本对齐
- 仍可继续核对：
  - compact 专用 cookie / header 行为
  - compact 失败时的 fallback 与日志诊断

不应继续补的部分：

- `thread/compact/start`
- 本地 compaction 事件流
- 客户端会话层的历史替换逻辑

原因：

- 这些是客户端 / 会话层职责，不是当前网关主线

建议优先级：

- 高

### P4. Responses WebSocket / prewarm / reuse 当前可暂缓

结论：

- 这三项不是当前主线必须做

原因：

- 当前产品主链是 HTTP + SSE 网关
- 官方这几项依赖 websocket 传输栈和连接状态机
- 在当前架构里，这不是“小补丁”，而是传输层重做

什么时候才应进入主线：

- 只有当目标变成“官方 `codex-rs` 新主线客户端不改配置直连 CodexManager”时

当前建议：

- 先暂缓
- 只在 P0/P1/P2/P3 收口后再决定

## 当前不应再继续推进的方向

以下内容不再作为当前对齐主线：

- `thread/*`
- `turn/*`
- `thread/realtime/*`
- `review/start`
- `skills/*`
- `plugin/*`
- `app/*`
- `config/*`
- `mcpServer/*`
- 本地 `/rpc/events` SSE 兼容层
- 本地 `GET /rpc` WebSocket app-server 兼容层

原因：

- 对当前产品没有直接用户价值
- 容易引入启动、RPC、状态管理复杂度
- 已经开始对桌面启动稳定性产生干扰

## 建议的重新起步顺序

1. 登录启动失败不再放行
2. callback 错误模型补齐
3. workspace 二次校验
4. refresh 失效判定从字符串匹配继续收紧
5. plan type 读取路径统一
6. `/v1/responses` 做一轮严格 on-wire 复核
7. `/v1/responses/compact` 做一轮专用诊断与失败分类收口
8. 最后再决定 websocket / prewarm / reuse 要不要进
