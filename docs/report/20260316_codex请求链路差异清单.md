# Codex 请求链路差异清单

生成时间：2026-03-17

## 范围

本清单只覆盖真正影响下面这条链路的部分：

- `Codex / Codex 风格客户端`
- `CodexManager`
- `chatgpt.com/backend-api/codex`

只看这些内容：

- 登录与 token
- `/v1/responses`
- `/v1/responses/compact`
- 请求头、压缩、会话锚点、失败诊断
- `responses websocket / prewarm / reuse` 是否值得进入主线

明确不看：

- `thread/*`
- `turn/*`
- `review/*`
- `skills/*`
- `plugin/*`
- `app/*`
- 本地 app-server / RPC 兼容层

## 当前可保留的已对齐项

### 登录与账号基础

- 浏览器登录主干参数已基本对齐：
  - `/oauth/authorize`
  - PKCE
  - `state`
  - `redirect_uri`
  - `scope=openid profile email offline_access api.connectors.read api.connectors.invoke`
  - `id_token_add_organizations=true`
  - `codex_cli_simplified_flow=true`
  - `allowed_workspace_id`
  - `originator`

- 授权码换 token 的请求体已对齐：
  - `grant_type=authorization_code`
  - `code`
  - `redirect_uri`
  - `client_id`
  - `code_verifier`

- `id_token -> openai-api-key` 的 token exchange 请求体已对齐
- refresh token 请求体已对齐官方 `application/json` 形状
- refresh / usage HTTP client 已挂上官方默认：
  - `Originator`
  - `User-Agent`
  - `x-openai-internal-codex-residency`
- `/oauth/token` 登录链路现在也已挂上：
  - `Accept: application/json`
  - `Originator`
  - `User-Agent`
  - `x-openai-internal-codex-residency`
- usage endpoint 请求已统一使用官方 `ChatGPT-Account-ID` 头语义
- `CPA no cookie` 模式现在只抑制 `Cookie` / 粘性相关头，不再错误连带去掉 `ChatGPT-Account-ID`
- usage endpoint 的 challenge / HTML / auth header 调试信息
  现在会走稳定摘要，不再只剩裸状态码或整页 HTML
- token endpoint 错误解析已贴近官方：
  - 优先读 `error_description`
  - 其次读嵌套 `error.message`
  - 再回退 `error code`
  - transport error 会做敏感 URL 脱敏
  - 遇到 challenge/HTML 页面时会摘要成：
    - `Cloudflare 安全验证页`
    - `上游返回 HTML 错误页`
- token endpoint / api key exchange 失败摘要现在也会带上游调试头：
  - `request_id`
  - `cf-ray`
  - `auth_error`
- refresh 失效判定已收紧到官方主干语义：
  - 仅 `401 Unauthorized` 进入 refresh 认证失败分支
  - `403`、challenge 页、代理异常不再误摘号
- refresh `401` 内部原因已收口到稳定枚举：
  - `refresh_token_expired`
  - `refresh_token_reused`
  - `refresh_token_invalidated`
  - `refresh_token_unknown_401`
- 401 refresh 错误文案已映射到官方主干消息：
  - `refresh_token_expired`
  - `refresh_token_reused`
  - `refresh_token_invalidated`
  - unknown 401
- `planType` 读取已优先按最新 access token claims 收口
- 对外 `planType` 已统一到官方枚举语义：
  - `free / go / plus / pro / team / business / enterprise / edu / unknown`
- 未知 plan 现在会额外保留原始值用于诊断，不再只剩统一的 `unknown`

- 默认 `issuer / client_id / originator` 基线已对齐官方默认值

### `/v1/responses`

- 普通 `/responses` 仍走 HTTP + SSE 主链路，和当前产品形态一致
- 已透传并维护这些关键会话信号：
  - `prompt_cache_key`
  - `session_id`
  - `x-client-request-id`
  - `x-openai-subagent`
  - `x-codex-beta-features`
  - `x-codex-turn-metadata`（ASCII 安全值）
  - `x-codex-turn-state`
- HTTP `/responses` 已不再显式发送这些非官方头：
  - `Conversation_id`
  - `OpenAI-Beta`
  - `Connection: Keep-Alive`
  - `Version`

- 模型列表 `/models` 请求已收回到同一套官方头基线：
  - `originator`
  - 动态 `User-Agent`
  - `ChatGPT-Account-ID`
  - `x-openai-internal-codex-residency`
  - 保留 `client_version` 查询参数
  - 不再显式发送 `Version`
  - challenge / HTML / auth header 调试信息会走稳定摘要，不再原样透传整页 body

- 已支持动态：
  - `Originator`
  - `User-Agent`
  - `x-openai-internal-codex-residency`

- 已支持对流式 `/responses` 做 `zstd` 请求体压缩
- 当 `conversation_id` 已存在且客户端未显式提供时，`prompt_cache_key` 会自动补齐到同一线程锚点
- 当客户端未显式提供 `x-client-request-id` 时，
  `/responses` 现在会回退到稳定的线程/session 锚点，
  更贴近官方 `conversation_id -> x-client-request-id` 的默认行为
- 当 `/responses` 已有 `prompt_cache_key` 线程锚点时，
  网关现在会优先让 `session_id / x-client-request-id` 也跟随该锚点，
  不再让旧 `Session_id` 抢占线程语义
- 当入站明确携带 `Conversation_id` 时，
  网关现在也会忽略旧的 `x-client-request-id`，让线程锚点覆盖旧 request id，
  更贴近官方 `ResponsesClient` 的覆盖顺序
- 当旧 `Session_id` 已被新的线程锚点覆盖时，
  网关现在也会丢弃旧的 `x-codex-turn-state`，避免把上游路由继续粘到历史 turn
- Codex `/responses` 请求体已进一步贴近官方默认值：
  - `tool_choice` 缺失时补成 `auto`
  - 存在 `reasoning` 时自动补齐 `include=["reasoning.encrypted_content"]`
  - 已移除非官方字段 `encrypted_content`

- 已有 challenge / 401 / 429 / failover / route quality / 请求日志落盘的主链路

### `/v1/responses/compact`

- 已走真实上游 `/responses/compact`
- 已按 compact 专用 allowlist 和头部语义发请求
- 已按非流式 JSON unary 请求处理
- 入站未显式携带 `x-openai-subagent` 时，
  compact 现在会默认补 `x-openai-subagent=compact`，贴近官方 `compact_remote` 行为
- compact 上游若返回 `2xx` 但 body 不是合法 `{"output":[...]}` 成功体，
  现在会改判为 `502`，不再把 HTML/challenge/异常 JSON 假成功透传给客户端
- compact 上游若返回 `403/5xx` 的 HTML/challenge 页，
  现在会改成结构化 JSON 错误返回，不再把整页 HTML 直接回给客户端
- compact challenge / HTML 结构化错误现在会带上游调试头摘要：
  - `request_id`
  - `cf-ray`
  - `auth_error`

### 失败诊断链路

- `gateway-trace.log` 现在会对最终失败的请求保留上下文缓冲：
  - `REQUEST_START`
  - `CANDIDATE_*`
  - `ATTEMPT_PROFILE`
  - `ATTEMPT_RESULT`
  - `BRIDGE_RESULT`
  - `FAILED_REQUEST`
- 最终成功的请求仍不会留下 trace 文件，避免把成功链路也写成排障日志
- `BRIDGE_RESULT` 和失败请求日志现在会额外带上游调试头摘要：
  - `request_id`
  - `cf-ray`
  - `content-type`
- 普通 `/responses` 的 challenge / HTML 失败日志现在也会带：
  - `request_id`
  - `cf-ray`
  - `auth_error`

## 当前主要差异

### P0. 登录主干已收口，剩余差异转到 token 刷新与错误分类

已完成：

- 登录启动不再失败放行
- callback 现在会处理：
  - `error`
  - `error_description`
  - `state` 缺失/失效
  - 缺少 `code`
- callback 成功后会按登录会话里的 `workspace_id` 做二次校验
- `login_sessions` 已持久化 `workspace_id`
- token endpoint 错误解析与 transport 脱敏已收口到接近官方

影响：

- 登录成功率和 workspace 绑定正确性已经明显收口
- 剩余问题主要转移到 refresh 错误分类与少量请求头同步链路

建议优先级：

- 已完成，后续只做微调

### P1. refresh / plan type 仍有少量误判空间

当前差异：

- 未知 `401` 目前仍会按永久失效处理，后续可再决定是否细分成 `Other`
- `401` reason 虽已结构化到内部枚举，但还没有独立持久化字段；当前仍通过错误链路传递

影响：

- 已基本消除 403/challenge 误摘号
- 剩余风险主要是未知 401 的保守处理，以及 reason 尚未单独落库
- 会间接影响模型选择、候选过滤和路由策略

建议优先级：

- 中高

### P2. `/v1/responses` 主链路还需要继续做 on-wire 复核

当前已做得比较接近，但仍应继续核对这些点：

- 请求体字段白名单与默认值的剩余边角
- stream / non-stream 分支上的 header profile
- cookie、turn state、conversation 相关头在不同链路上的带法
- 失败重试、failover、日志落盘时机

说明：

- 当前不是“完全没对齐”
- 而是还缺一轮严格的 on-wire 收口

影响：

- 直接影响成功率
- 直接影响 challenge 命中概率
- 也会影响日志可解释性

建议优先级：

- 高

### P3. `/v1/responses/compact` 主体应保留，但只补远端压缩真正相关部分

当前差异：

- compact 请求本身已基本对齐
- 仍可继续核对：
  - compact 专用 cookie / header 行为
  - compact 失败时的 fallback 与日志诊断细分

不应继续补的部分：

- `thread/compact/start`
- 本地 compaction 事件流
- 客户端会话层的历史替换逻辑

原因：

- 这些是客户端 / 会话层职责，不是当前网关主线

建议优先级：

- 高

### P4. Responses WebSocket / prewarm / reuse 当前可暂缓

结论：

- 这三项不是当前主线必须做

原因：

- 当前产品主链是 HTTP + SSE 网关
- 官方这几项依赖 websocket 传输栈和连接状态机
- 在当前架构里，这不是“小补丁”，而是传输层重做

什么时候才应进入主线：

- 只有当目标变成“官方 `codex-rs` 新主线客户端不改配置直连 CodexManager”时

当前建议：

- 先暂缓
- 只在 P0/P1/P2/P3 收口后再决定

## 当前不应再继续推进的方向

以下内容不再作为当前对齐主线：

- `thread/*`
- `turn/*`
- `thread/realtime/*`
- `review/start`
- `skills/*`
- `plugin/*`
- `app/*`
- `config/*`
- `mcpServer/*`
- 本地 `/rpc/events` SSE 兼容层
- 本地 `GET /rpc` WebSocket app-server 兼容层

原因：

- 对当前产品没有直接用户价值
- 容易引入启动、RPC、状态管理复杂度
- 已经开始对桌面启动稳定性产生干扰

## 建议的重新起步顺序

1. `/v1/responses` 做一轮严格 on-wire 复核
2. `/v1/responses/compact` 做一轮专用诊断与失败分类收口
3. token endpoint 401 reason 继续细分
4. 失败诊断链路继续增强
5. 最后再决定 websocket / prewarm / reuse 要不要进
