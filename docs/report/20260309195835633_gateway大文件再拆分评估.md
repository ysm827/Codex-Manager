# gateway 大文件再拆分评估

## 范围

本报告评估以下三个高风险文件的下一步拆分方向：

- `crates/service/src/gateway/observability/http_bridge.rs`（约 2862 行）
- `crates/service/src/gateway/protocol_adapter/request_mapping.rs`（约 1443 行）
- `crates/service/src/gateway/protocol_adapter/response_conversion/sse_conversion.rs`（约 976 行）

## 1. `http_bridge.rs`

### 当前职责

当前文件同时承担：

- 下游 HTTP 响应桥接
- SSE / 非 SSE 结果写回
- `output_text` 截断与 usage 聚合
- 上游错误提示解析
- trace / 头部补写
- 响应传输异常归因

### 建议拆分

建议拆成以下子模块：

- `bridge_result.rs`
  - `UpstreamResponseUsage`
  - `UpstreamResponseBridgeResult`
  - usage merge / signal 判断
- `output_text.rs`
  - `output_text` 限制读取
  - 文本截断与 marker 逻辑
- `sse_reader.rs`
  - SSE 逐行读取
  - terminal marker 判定
  - event 级 usage 抽取
- `json_reader.rs`
  - 非流式 JSON usage / error hint 抽取
- `response_writer.rs`
  - tiny_http `Response` 写回
  - trace / error header 注入

### 推荐迁移顺序

1. 先抽纯数据结构与帮助函数：`bridge_result.rs`、`output_text.rs`
2. 再拆流式读取：`sse_reader.rs`
3. 再拆非流式 JSON：`json_reader.rs`
4. 最后把写回与头部拼装抽到 `response_writer.rs`

### 风险点

- SSE terminal 判定很容易在拆分时漏掉 `response.completed` / `[DONE]` 等兼容分支
- `delivery_error` 与 `stream_terminal_error` 的优先级不能漂移
- `output_text` 限长要保持当前字节级行为，不可变成字符级

## 2. `request_mapping.rs`

### 当前职责

当前文件同时承担：

- OpenAI Chat 输入映射
- OpenAI Responses 输入映射
- Anthropic 输入兼容
- 工具名缩短与恢复 map
- reasoning / model / instruction 默认值
- 消息内容归一化

### 建议拆分

建议按“协议来源 + 通用工具”拆成：

- `request_mapping/openai_chat.rs`
  - chat/completions -> 内部请求映射
- `request_mapping/openai_responses.rs`
  - responses -> 内部请求映射
- `request_mapping/anthropic.rs`
  - anthropic 风格请求映射
- `request_mapping/tools.rs`
  - tool 名缩短
  - tool restore map
  - tool_choice / tool_calls 归一化
- `request_mapping/message_content.rs`
  - content 提取与 role 归一化
- `request_mapping/defaults.rs`
  - 默认 model / reasoning / instruction 常量与默认值策略

### 推荐迁移顺序

1. 先抽 `tools.rs` 与 `defaults.rs`
2. 再拆 `message_content.rs`
3. 再把 OpenAI chat / responses 各自外提
4. 最后保留一个薄的 `request_mapping.rs` 作为总入口

### 风险点

- `tools` 和 `tool_calls` 相关逻辑是高兼容敏感区，拆分后必须继续保持统一 restore map
- OpenAI chat 与 responses 之间共享的消息归一化逻辑不能出现两套实现漂移
- reasoning / default model 不能因协议不同而无意改变默认值

## 3. `sse_conversion.rs`

### 当前职责

当前文件同时承担：

- Anthropic JSON -> SSE
- OpenAI / Responses SSE 事件拼装
- tool input 增量 JSON 序列化
- reasoning / thinking block 增量转换
- finish reason / usage 事件补齐

### 建议拆分

建议按“适配目标 + 事件工具”拆成：

- `response_conversion/sse_events.rs`
  - 通用 `append_sse_event`
  - 事件级 JSON 包装
- `response_conversion/sse_anthropic.rs`
  - Anthropic JSON -> SSE
- `response_conversion/sse_openai_chat.rs`
  - OpenAI Chat 流式输出
- `response_conversion/sse_responses.rs`
  - Responses 流式输出
- `response_conversion/sse_tool_delta.rs`
  - tool input partial JSON
  - function call 参数增量
- `response_conversion/sse_reasoning.rs`
  - thinking / reasoning 增量与签名块

### 推荐迁移顺序

1. 先抽 `sse_events.rs`
2. 再抽 `sse_tool_delta.rs`
3. 再抽 `sse_reasoning.rs`
4. 最后按适配目标拆 `sse_anthropic.rs` / `sse_openai_chat.rs` / `sse_responses.rs`

### 风险点

- tool delta 与 reasoning delta 事件顺序不能变
- Anthropic thinking/signature 事件映射是高兼容敏感区
- stream 结束时的 finish reason、usage 结尾事件必须保持当前行为

## 建议优先级

建议按以下顺序推进：

1. `request_mapping.rs`
2. `sse_conversion.rs`
3. `http_bridge.rs`

原因：

- `request_mapping.rs` 与 `sse_conversion.rs` 的职责边界更清晰，先拆收益更高
- `http_bridge.rs` 牵涉传输、写回、日志与错误归因，回归风险最大，适合最后拆

## 最低验证要求

涉及上述任一文件拆分时，至少补：

- `cargo test -p codexmanager-service`
- `scripts/tests/gateway_regression_suite.ps1`
- `/v1/chat/completions` stream / non-stream
- `/v1/responses` stream / non-stream
- tools / `tool_calls` 兼容验证
