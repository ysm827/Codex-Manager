# 脚本与发布职责对照

## `.github/actions/` 与 `scripts/release/` 对照表

| 场景 | GitHub Action | 对应脚本/职责 |
|---|---|---|
| 版本校验 | `validate-release-version` | `scripts/release/assert-release-version.ps1` |
| Tauri 构建重试 | `build-tauri-with-retry` | `scripts/release/build-tauri-with-retry.ps1` / `.sh` |
| Service 打包 | `stage-service-package` | `scripts/release/stage-service-package.ps1` / `.sh` |
| GitHub Release 发布 | `publish-github-release` | `scripts/release/publish-github-release.sh` |
| release 环境准备 | `setup-release-env` | workflow 内环境装配，当前没有完全等价的独立顶层脚本 |
| 前端 dist 准备 | `prepare-frontend-dist` | workflow 内构建/下载前端产物收口，当前没有单独顶层脚本 |

## 边界约定

### `.github/actions/`

负责：

- workflow 内可复用步骤
- 跨 job 统一输入输出
- CI / Release 环境封装

### `scripts/release/`

负责：

- 实际构建、打包、发布动作
- 被本地或 workflow 调用的脚本实现
- 尽量保持可独立执行

### `scripts/*.ps1|*.sh`

负责：

- 面向开发者的顶层入口
- 统一参数组织
- 将复杂 release 子步骤收口

## 当前历史脚本盘点

### 仍有保留价值

- `rebuild.ps1`
- `bump-version.ps1`
- `tests/gateway_regression_suite.ps1`
- `tests/chat_tools_hit_probe.ps1`
- `tests/codex_stream_probe.ps1`
- `release/*`

原因：

- 仍被本地开发、发布链路或 workflow 实际使用
- 不是孤立遗留文件

### 当前未发现“只剩单点用途且可直接删除”的脚本

现阶段更合适的策略不是删除，而是：

1. 用 `scripts/README.md` 标明入口分层
2. 把 CI 专用脚本与本地入口分开描述
3. 后续如果某个脚本连续多个版本无人引用，再考虑归档或合并

## 维护建议

- 新增 workflow 能力时，优先判断是否应复用现有 action
- 新增发布脚本时，先决定它是“顶层入口”还是“CI 子步骤”
- 若脚本不适合本地执行，要在 README 中明确标成“仅 CI”
