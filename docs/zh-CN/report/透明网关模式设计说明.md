# 透明网关模式设计说明

## 1. 背景

当前 CodexManager 已经具备：

- 多账号管理
- 用量统计与审计
- 请求日志与可观测性
- 失败切换 / 会话亲和 / 路由策略
- OpenAI / Anthropic / Gemini / Codex 等多协议适配

这些能力解决了大量实际问题，但也带来了一个越来越明显的产品边界问题：

- 对 `Codex` 原生流量来说，网关不仅在做“账号替换与路由”，还承担了较多“请求重写 / 参数补全 / 协议兼容 / 输出适配”的职责。
- 用户会逐渐感觉自己在使用“Codex 的替代层”，而不是“Codex 的账号池与透明网关”。

如果项目的主目标仍然是：

- `Codex` 账号池
- `Codex` 透明网关
- `Codex` 请求审计与管理壳

那么默认产品哲学需要向“最小侵入、最大透传”收敛。

---

## 2. 目标

当前实现目标是：

1. 对 `Codex` 原生请求尽量保持原样透传。
2. 网关默认只承担账号选择、鉴权替换、路由与观测职责。
3. 把“增强兼容 / 协议改写 / 主观补全”从默认路径中拿出来，降级为可选能力。
4. 让用户明确感知：自己仍在使用 `Codex`，CodexManager 只是管理层与流量调度层。

一句话概括：

> 对 Codex 原生流量，默认只换账号，不改协议语义。

---

## 3. 非目标

透明网关模式不是为了：

- 删除现有所有协议适配能力
- 放弃 OpenAI / Anthropic / Gemini 等兼容入口
- 取消故障切换、重试、限流、日志等网关能力
- 追求与 Codex 源码逐字节完全相同的内部实现

透明网关模式只约束一件事：

- 对已经是 `Codex` 原生形态的请求，不再默认做“主观增强式重写”。

---

## 4. 设计原则

### 4.1 原生请求优先

如果入站请求已经是 `Codex` 原生协议形态，则默认走透明路径，而不是兼容适配路径。

### 4.2 最小必要改动

默认只允许以下四类改动：

1. 鉴权替换
2. 必要的会话亲和字段传递
3. 路由与 failover 所需的最小上下文补充
4. 日志、审计、观测相关的内部处理

### 4.3 不替用户做主

透明模式下，网关不应主动替用户决定：

- `model`
- `reasoning`
- `tool_choice`
- `parallel_tool_calls`
- `instructions`
- `include`
- `service_tier`
- `prompt_cache_key`

除非：

- 上游协议强制要求
- 安全 / 合规必须
- 请求已经无效且需要最小修复才能通过上游校验
- 用户明确开启增强模式

### 4.4 协议适配和原生直通分层

当前产品不再对用户暴露“透明 / 增强”切换。

- `Codex` 原生请求固定走原生直通路径
- `Claude / Gemini` 继续走协议适配路径
- 是否属于“适配入口”由客户端协议决定，而不是让用户在设置里手动切模式

---

## 5. 透明网关模式的默认行为

### 5.1 请求头

对于 `Codex` 原生请求：

- 保留原始业务语义头
- 过滤掉不应直接透传的鉴权字段
- 用目标账号的 `access_token` / 会话上下文替换认证信息
- 保留必要的会话亲和字段，如：
  - `x-client-request-id`
  - `session_id`
  - `x-codex-turn-state`
  - `x-codex-turn-metadata`

透明模式下，请求头的核心原则是：

- 不主动新增“非必须语义头”
- 不主动删除“上游依赖的原始头”
- 不把内部调度字段泄漏成上游协议字段

### 5.2 请求体

对于 `Codex` 原生 `responses` / `chat.completions` 请求：

- 默认不改字段语义
- 默认不补默认值
- 默认不做模型替换
- 默认不做工具定义重映射
- 默认不做 reasoning 自动推断

仅允许做：

- 上游明确要求的最小修复
- 路由层必要的内部标记剥离
- 非协议字段移除

### 5.3 路由与账号池

透明模式下保留以下能力：

- 多账号选择
- `ordered / balanced` 路由策略
- 单账号并发上限
- failover
- cooldown
- 账号状态过滤
- 会话亲和

这些能力不会改变 Codex 协议本身，属于透明网关的核心价值。

### 5.4 可观测性

透明模式下继续保留：

- request log
- trace log
- 上游响应状态记录
- token / usage 统计
- 请求路径与适配结果记录

日志里只需要继续保留请求路径、协议适配结果和上下游信息，便于区分：

- 原生 `Codex` 请求
- `Claude / Gemini` 等适配入口

---

## 6. 哪些能力属于透明层，哪些属于增强层

### 6.1 透明层

这些能力应始终保留，且默认启用：

- 账号池管理
- token 替换
- 请求路由
- 会话亲和
- failover / retry
- 审计日志
- 监控指标
- 限流与并发保护
- 上游超时控制

### 6.2 增强层

这些能力原则上不应默认套在 Codex 原生请求上：

- OpenAI / Anthropic / Gemini 协议互转
- `tools` 结构重写
- `tool_choice` 自动收敛
- `reasoning` 默认补全
- `parallel_tool_calls` 自动补值
- `instructions` 缺省补空
- `include` 自动补 reasoning 字段
- `prompt_cache_key` 主观改写
- 输出结构二次适配
- SSE / WS 的兼容桥接包装

### 6.3 历史遗留层

以下能力需要逐项审视是否仍应保留为默认行为：

- 为“提高成功率”而引入的隐式请求修复
- 为兼容旧客户端而遗留的 body 重写逻辑
- 与 Codex 原生路径无关，但目前混入统一发送链路的补丁

这些逻辑应逐步迁移到：

- 显式增强模式
- 协议适配专用入口
- 或直接删除

---

## 7. 推荐的产品形态

前端 / 设置项不再暴露“透明 / 增强”模式切换，而是固定采用下面这套语义：

### 7.1 Codex 原生直通

- 对 `Codex` 原生请求最小修改
- 只做账号替换、路由、日志与审计

### 7.2 Claude / Gemini 协议适配

- 保留 `Anthropic / Gemini` 到内部 `Responses` 的协议映射
- 继续保留响应回写和流式桥接
- 不把这条兼容链路包装成可切换的“增强模式”

---

## 8. 实施路线

### 阶段 1：边界定义

目标：

- 明确“什么叫 Codex 原生请求”
- 明确“原生直通”和“协议适配”的边界
- 给关键链路打上模式标记

产物：

- 模式枚举
- 文档
- trace / request log 中的模式字段

### 阶段 2：默认路径收敛

目标：

- 对 Codex 原生请求默认走透明路径
- 把主观补全从默认链路中剥离

优先处理项：

- `reasoning` 自动补全
- `tool_choice` 自动收敛
- `parallel_tool_calls` 自动补值
- `instructions` 缺省注入
- `include` 自动补齐

### 阶段 3：增强逻辑显式化

目标：

- 把兼容与增强能力挂到显式模式下
- 日志中可看到“哪条增强逻辑介入了”

### 阶段 4：前端与文案收敛

目标：

- 页面文案从“更智能代理”收回到“透明网关 + 账号池”
- 让用户明确知道默认推荐的是透明模式

---

## 9. 验收标准

透明网关模式上线后，应满足：

1. 原生 Codex 请求默认不再经过主观语义改写。
2. 请求头与请求体的出站结果，在协议层面与 Codex upstream 保持同形。
3. 路由、账号替换、日志、审计、失败切换仍然正常工作。
4. trace / request log 中能明确看出本次请求是否使用透明模式。
5. 增强逻辑必须可以被显式关闭，并与透明模式隔离。

---

## 10. 风险

### 10.1 成功率短期波动

去掉某些历史上“为了成功率而做的自动补全”后，短期内可能暴露更多上游真实校验错误。

这是预期内现象，不应再靠默认隐式补丁掩盖。

### 10.2 兼容路径与透明路径耦合过深

当前请求改写、协议适配、上游发送链路可能已经有较多共享逻辑。拆分时需要注意：

- 不要让透明路径继续偷偷复用增强逻辑
- 不要让增强路径被透明路径回归打断

### 10.3 用户认知迁移

一部分用户已经习惯“网关会帮忙补字段 / 修请求”的行为。切到透明模式后，需要有更清晰的设置说明与日志反馈。

---

## 11. 最终结论

CodexManager 的主航向应该回到：

> 透明网关 + 账号池管理 + 审计观测

而不是：

> Codex 的替代品

默认推荐的“透明网关模式”应该成为系统的黄金路径；增强兼容能力继续保留，但应退到明确可选的位置。

这样既能保住项目的核心价值，也能避免产品边界继续向“替代 Codex”方向漂移。
