# Codex请求与返回解析差异清单-2026-04-19

## 1. 范围

本文只比较和 `responses` 主链路直接相关的差异，范围包括：

- 官方源码：`D:\MyComputer\own\GPTTeam\codex\codex`
- 当前网关：`D:\MyComputer\own\GPTTeam\CodexManager\CodexManager`
- 关注路径：原生 `/v1/responses`、`/v1/responses/compact`、以及 `OpenAI 兼容入口 -> /v1/responses` 适配链路

## 2. 结论先行

- 原生 `/v1/responses -> Codex backend` 这条主链路，当前已经做到“请求头高度对齐、请求体 shape 高度对齐、SSE 解析组件对齐”。
- 现在最值得继续改的，不是大部分请求头，而是少数尾差：
  - 缺失 `instructions` 时我们会主动补空字符串
  - `chat/completions` 和 `completions` 仍然要先适配成内部 `responses`
  - `/v1/responses` 虽然已改用 `eventsource-stream`，但仍然是“桥接层解析后再重放”，不是官方那种直接消费 typed `ResponseEvent`
- 如果目标是“优先降低报错概率”，优先改请求 shape 尾差即可。
- 如果目标是“尽可能逐行为贴近官方 Codex”，下一阶段要继续收 transport 和事件模型这两类差异。

## 3. 已对齐的关键项

| 项目 | 官方 Codex | 当前网关 | 当前判断 |
| --- | --- | --- | --- |
| `/v1/responses` 请求头骨架 | `Authorization`、`ChatGPT-Account-ID`、`Content-Type`、`Accept: text/event-stream`、`User-Agent`、`originator`、`session_id`、`x-codex-window-id`、`x-client-request-id` | 已按同类字段构造，且优先保留来路 `originator` / `User-Agent` | 基本一致 |
| `/v1/responses/compact` 头部语义 | `Accept: application/json`，不带 `x-client-request-id` | 已按 compact 独立头策略构造 | 基本一致 |
| `parallel_tool_calls` 默认值 | 官方请求结构为必填布尔值 | 兼容改写路径缺失时会补 `false` | 已对齐 |
| `include` 默认值 | 官方请求结构为 `Vec<String>` | 兼容改写路径缺失时会补空数组；有 reasoning 时会补 reasoning include | 已对齐 |
| 请求体压缩 | 官方支持 `zstd` 请求压缩 | 当前上游发送链路也支持 `zstd` 并自动补 `Content-Encoding: zstd` | 已对齐 |
| `/v1/responses` SSE 解析组件 | `bytes_stream() + eventsource-stream` | 现在也改成 `eventsource-stream` | 已对齐到组件层 |

## 4. 仍有差异

| 优先级 | 环节 | 官方 Codex | 当前网关 | 影响 | 建议 |
| --- | --- | --- | --- | --- | --- |
| P0 | `instructions` 缺失时的行为 | 官方 `ResponsesApiRequest.instructions` 带 `skip_serializing_if = "String::is_empty"`，空字符串不会发出去 | 当前兼容改写会主动补 `instructions: ""`，并原样发到上游 | 请求 shape 仍非逐字段一致；可能影响上游启发式或更严格校验 | 去掉自动补空串；缺失时直接省略字段 |
| P1 | `OpenAI compat` 入口路径 | 官方原生客户端直接请求 `/v1/responses` | 当前 `/v1/chat/completions`、`/v1/completions` 先适配成内部 `/v1/responses`，再把响应适配回原协议 | 多了一层 request/response 映射，兼容链路比原生链路更复杂 | 如果前端或适配器可控，尽量优先直接走原生 `/v1/responses` |
| P1 | `/v1/responses` 响应事件模型 | 官方把 SSE 事件解析为 typed `ResponseEvent`，后续逻辑基于事件枚举工作 | 当前虽然用了 `eventsource-stream`，但仍在桥接层把事件重新拼成 SSE 字节，并用通用 collector 做 usage / terminal 聚合 | 语义已接近，但不是 typed event 级别的一致；观测和错误归因仍可能丢细节 | 如果要继续对齐，可在观测层增加 typed `ResponseEvent` 映射，而不是只做通用 SSE 聚合 |
| P2 | 上游 transport 组件 | 官方是 `ReqwestTransport + CodexHttpClient` 的 async transport | 当前网关仍是 `reqwest::blocking::Client` + 自己的 proxy / bridge 管线 | 连接、超时、错误传播、桥接行为不可能与官方完全等价 | 仅在追求“尽量逐行为一致”时再考虑重构；不是当前最高收益项 |
| P2 | SSE 回放方式 | 官方内部直接消费事件流，不需要把事件重新编码回 SSE | 当前 `/v1/responses` passthrough 会把 `eventsource-stream` 解析后的事件重新编码成 SSE 帧，再输出给下游 | 大多数情况下没问题，但字节级不可能完全一致 | 如果后续要做到极限对齐，可研究“原始字节直通 + sidecar 观测”的双通路方案 |
| P2 | 错误模型 | 官方会把 `response.failed` / `response.incomplete` 等事件收束到自己的 `ApiError` 模型 | 当前网关除了事件内错误，还会叠加本地 bridge 错误分类、toast 友好化和 trace 日志 | 用户看到的错误文本和官方不可能完全同文案 | 如果重点是“问题更好排查”，继续强化 `trace_id`、upstream hint、stage 标识即可，不必强求完全同文案 |

## 5. 目前不建议再动的项

下面这些点已经不属于当前的主要风险源，短期内不建议重复折腾：

| 项目 | 原因 |
| --- | --- |
| `originator` / `User-Agent` | 当前已优先保留来路 Codex 身份，不再强行覆盖为本地固定值 |
| `parallel_tool_calls` / `include` | 已补成更接近官方结构的默认 shape |
| `zstd` 压缩 | 当前发送链路已支持，不是主要差异点 |
| `/v1/responses` SSE 组件选择 | 已从手搓解析切到 `eventsource-stream`，继续换组件的收益已经不高 |

## 6. 建议改动顺序

| 顺序 | 建议改动 | 预期收益 | 风险 |
| --- | --- | --- | --- |
| 1 | 去掉缺失 `instructions` 时的空字符串注入 | 让请求体更接近官方逐字段 shape，改动小、收益直接 | 低 |
| 2 | 对我们自己可控的调用方，优先切到原生 `/v1/responses` | 缩短兼容适配链路，减少一层协议映射 | 中 |
| 3 | 给 `/v1/responses` 观测链路增加 typed `ResponseEvent` 解析 | 让错误归因、usage 统计、事件审计更贴近官方 | 中 |
| 4 | 评估是否值得把上游 transport 进一步往 async 官方语义靠拢 | 只有在追求“极限一致性”时才有意义 | 高 |

## 7. 证据锚点

### 官方源码

- `codex-rs/codex-api/src/common.rs`
  - `ResponsesApiRequest` 结构里：
    - `instructions` 为空时跳过序列化
    - `parallel_tool_calls` 是布尔必填
    - `include` 是数组必填
- `codex-rs/codex-api/src/endpoint/responses.rs`
  - `/responses` 流式请求使用 `Accept: text/event-stream`
- `codex-rs/codex-client/src/transport.rs`
  - 上游流读取使用 `bytes_stream()`
  - 请求压缩支持 `zstd`
- `codex-rs/codex-api/src/sse/responses.rs`
  - SSE 解析核心是 `stream.eventsource()`

### 当前网关

- `crates/service/src/gateway/request/request_rewrite_responses.rs`
  - `/v1/responses` 的 body 兼容改写逻辑
  - 当前仍会补 `instructions: ""`
- `crates/service/src/gateway/protocol_adapter/codex_adapter.rs`
  - `chat/completions` / `completions` -> `/v1/responses` 的入口适配
- `crates/service/src/gateway/upstream/headers/codex_headers.rs`
  - Codex upstream 头构造
- `crates/service/src/gateway/upstream/attempt_flow/transport.rs`
  - 当前上游发送链路
  - `zstd` 请求压缩
- `crates/service/src/gateway/observability/http_bridge/stream_readers/openai_responses.rs`
  - `/v1/responses` 已切到 `eventsource-stream`
  - 但当前仍是桥接层解析后再回放 SSE
- `crates/service/src/gateway/observability/http_bridge/delivery.rs`
  - `/v1/responses` passthrough 分支选择 `OpenAIResponsesPassthroughSseReader`

## 8. 一句话建议

如果当前目标是“先把最容易出问题的差异收掉”，下一步优先改 `instructions` 空字段注入。  
如果目标是“尽量做到像官方一样”，那就把 `/v1/responses` 继续往 typed event 和 transport 语义收，但这已经是第二阶段工作。
