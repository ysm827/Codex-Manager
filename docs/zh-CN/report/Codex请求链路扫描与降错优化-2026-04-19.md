## 已落地优化

### 1. 修复 ChatGPT challenge 补救分支失效

问题：

- 服务端原本设计了 “遇到 challenge 后，最后一个候选账号可尝试去掉压缩后重试一次” 的补救链路。
- 但 `process_upstream_post_retry_flow` 的判断条件写反，导致这条分支实际上不会进入。

影响：

- 当只剩最后一个候选账号时，请求更容易直接报错，而不是走一次低成本补救。
- 这会放大 Cloudflare / challenge 类错误在用户侧的出现频率。

处理：

- 修正 challenge 判断条件。
- 增加单测，覆盖 “最后一个候选账号挑战后恢复成功” 的场景。

### 2. Web RPC 非 2xx 错误信息补全

问题：

- `postJsonRpc` 之前遇到非 2xx 响应只抛出 `RPC 请求失败（HTTP xxx）`。
- 服务端如果已经返回结构化错误、`errorCode` 或 `traceId`，前端并没有取出来。

影响：

- 用户只能看到 HTTP 状态码，难以区分是鉴权、超时、后端不可达还是业务错误。
- 排障时需要额外看日志或抓包，前端提示价值偏低。

处理：

- 非 2xx 时优先读取响应体并抽取业务错误信息。
- 同时拼上 `X-CodexManager-Error-Code` 与 `X-CodexManager-Trace-Id`，便于快速对日志定位。

## 还值得继续做的优化

### 1. 收敛流式中断错误文案

- 目前前端已经收敛了 `request or response body error`、`stream read failed`。
- 但 `stream idle timeout` 仍可能直接暴露给用户。
- 建议继续把这类错误统一映射成更稳定的用户文案，并在服务端 `error_codes` 中补上明确分类。

### 2. 为 challenge / 429 / timeout 增加分场景统计

- 现在已经有 request log 和 gateway error log，但前端感知仍偏结果态。
- 建议把 challenge、rate limit、stream interrupted、timeout 单独聚合到 dashboard 指标里。
- 这样能更快判断是账号质量问题、网络波动还是协议兼容问题。

### 3. 评估 Web RPC 自动重试策略

- `fetchWithRetry` 当前默认会对 5xx / 429 自动重试。
- 对查询类 RPC 很友好，但对部分写操作会放大重复请求风险。
- 建议后续按命令粒度区分是否允许重试，避免“第一次成功但响应丢失，第二次重复提交”的边缘问题。

### 4. 让前端更多展示 trace id

- 这次已经把 trace id 带进错误文案。
- 后续可以考虑在请求日志页或 toast 展开区直接显示 trace id，减少用户反馈时的信息缺失。
