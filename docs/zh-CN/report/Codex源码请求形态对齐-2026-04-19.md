本次对照 `D:\MyComputer\own\GPTTeam\codex\codex` 最新源码后，可以确认 Codex 真正发往上游的主链路是 Rust `codex-rs/core -> codex-rs/codex-api` 的 `responses` HTTP SSE 请求，不是 TUI 自己拼 HTTP。

## 官方 Codex 最终请求形态

### 1. 标准 `/responses`

官方源码位置：

- `codex-rs/core/src/client.rs`
- `codex-rs/codex-api/src/endpoint/responses.rs`
- `codex-rs/codex-api/src/common.rs`

确认到的请求头：

- `Authorization: Bearer ...`
- `ChatGPT-Account-ID: ...`（仅 ChatGPT 账号态）
- `Content-Type: application/json`
- `Accept: text/event-stream`
- `User-Agent: <originator>/<version> (...)`
- `originator: <runtime originator>`
- `x-client-request-id: <conversation_id>`
- `session_id: <conversation_id>`
- `x-codex-window-id: <conversation_id>:<window_generation>`
- `x-openai-subagent: ...`（子代理时）
- `x-codex-parent-thread-id: ...`（thread spawn 子代理时）
- `x-codex-turn-metadata: ...`（有 turn metadata 时）
- `x-codex-turn-state: ...`（同一 turn 内 sticky routing 时）
- `x-codex-beta-features: ...`（启用 beta feature 时）
- `Content-Encoding: zstd`（仅 ChatGPT backend SSE 且压缩开启时）

明确不会带的头：

- `conversation_id`
- HTTP `/responses` 上不会主动加 `OpenAI-Beta`
- HTTP `/responses` 上不会主动加 `x-responsesapi-include-timing-metrics`

确认到的请求体字段：

- `model`
- `instructions`
- `input`
- `tools`
- `tool_choice`
- `parallel_tool_calls`
- `reasoning`
- `store`
- `stream`
- `include`
- `service_tier`
- `prompt_cache_key`
- `text`
- `client_metadata`

其中官方 Rust 结构有几个很容易被忽略的细节：

1. `include` 不是 `Option`，所以即使没有 reasoning，也会是 `[]`
2. `parallel_tool_calls` 是 `bool`，不是可选字段
3. `instructions` 是 `String`，空字符串时会被 `skip_serializing_if = "String::is_empty"` 省略
4. HTTP `/responses` 的 `client_metadata` 默认至少会带 `x-codex-installation-id`

## 这次对照后仍保留的差异

这些还没有在本次提交里强行改掉，主要是因为会牵涉现有兼容行为，建议后续按风险继续推进：

1. `instructions`
   - 官方空字符串会省略
   - 我们兼容改写路径仍会在缺失时补空字符串
2. `tool_choice`
   - 官方 Codex Rust 请求体当前是字符串字段
   - 我们兼容层仍允许保留显式 function object
3. `client_metadata`
   - 官方 HTTP `/responses` 默认会带 `x-codex-installation-id`
   - 我们当前是“来路有就保留，没有就不额外伪造”
