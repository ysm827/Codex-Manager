# release-all 旁路说明

## 目标

本文档说明 `release-all.yml` 在什么场景下可以走“旁路模式”，以及旁路时仍然需要保留哪些质量约束。

## 所谓旁路

这里的“旁路”不是绕开发布，而是：

- 仍然走同一个 workflow
- 但把 `run_verify` 调成 `false`
- 允许构建阶段回退到任务内本地前端构建
- 适用于紧急修复发版或验证门暂时不可用的场景

## 允许使用的场景

- 紧急修复需要尽快重新发版
- 预构建前端工件不可用，但代码本身已确认可构建
- 需要复跑同一 tag，仅修正 release 元数据或平台构建波动

## 不建议使用的场景

- 大版本发布
- 协议适配刚发生较大改动
- workflow 本身正在调整
- 本地尚未完成基本测试

## 旁路前最低检查

1. `pnpm -C apps run build`
2. `pnpm -C apps run test`
3. `cargo test --workspace`
4. 若涉及网关协议：`scripts/tests/gateway_regression_suite.ps1`

## 风险

- 跳过 workflow 内部分验证门
- 更依赖本地已完成的手工验证
- 若前端或脚本缓存脏了，问题会更晚暴露到 release 阶段

## 当前实现约束

- `run_verify=false` 时，workflow 会尝试在任务内回退构建前端
- 不会额外创建新的发布入口，仍维持 `release-all.yml` 为唯一统一入口
- Release 的 pre-release 判定仍由 `tag` 与 `prerelease` 输入共同决定

## 结论

可以旁路，但只能把它当紧急兜底，不应成为日常发版默认路径。
